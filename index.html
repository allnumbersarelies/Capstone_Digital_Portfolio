01 · The Problem

Why Standard Accuracy
Completely Misses the Point

Financial fraud is rare by design — which is exactly what makes it so hard to catch. A model that labels every single transaction as legitimate can score 99%+ accuracy. And it will catch exactly zero fraud.

What 99% Accuracy Looks Like

Legitimate Transactions 99.5%
Legitimate
Fraudulent Transactions 0.5%
Fraud
The Trap: If your model just says "legitimate" every time, it gets 99.5% accuracy — but catches zero fraud. This is why accuracy is the wrong metric entirely.
🎯

Recall — Did we catch the fraud?

Of all actual fraud cases, what fraction did our model correctly identify? A high-recall model finds most fraud, even if it also flags some legitimate transactions.

⚖️

Precision — Are our flags accurate?

Of all transactions we flagged as fraud, how many actually were? Low precision means fraud analysts are buried in false alarms and customer experience suffers.

📈

ROC AUC — Can the model rank risk?

Measures the model's ability to correctly rank fraud cases above legitimate ones across all possible decision thresholds. The industry-standard evaluation metric.

05 · Key Finding

The Sub-0.5 AUC:
An Honest Diagnostic

0.49

The Models Are Actually Ranking Fraud Backwards

On the Claude dataset, all three models produced AUC scores below 0.5 — meaning their probability rankings are inverted. The transactions they rate as most suspicious are actually slightly more likely to be legitimate, and vice versa. This is not a failure of the algorithms. It is a direct, honest measurement of how difficult the problem is.

Why this matters: If Glacier Bancorp used these scores to prioritize analyst review — checking the top N "most suspicious" transactions first — they would be looking at the wrong transactions. This finding would be invisible if you only looked at accuracy or recall. Reporting it openly is what makes the work trustworthy.
0.484
Claude Dataset — MLP (best of three)
Below 0.5: Model ranks fraud cases lower than legitimate ones in its internal confidence scores. Weak, noisy signal from intentional noise injection overwrites any pattern the model tries to find.
0.619
ChatGPT Dataset — MLP (best of three)
Above 0.5: Despite having only 10 fraud cases in the test set, the simpler, cleaner feature structure produces coherent probability rankings. Less noise = more reliable signal.
The Central Lesson: More data, more features, and a higher fraud rate do not automatically produce a better model. When additional features carry no incremental fraud signal — and are loaded with noise — they actively invert the model's confidence. Complexity without signal is just noise.
06 · Recommendations

Four Things Every Fraud
Detection System Should Do

01

Never Evaluate on Accuracy Alone

A 99.5%-accurate model can catch zero fraud. Any fraud system must be evaluated on recall, precision, F1, and ROC AUC — not headline accuracy.

02

Always Generate ROC Curves

The sub-0.5 AUC on the Claude dataset was completely invisible from the confusion matrix. ROC curves reveal whether a model's scores can be trusted for prioritization.

03

Address Class Imbalance Deliberately

SMOTE, cost-sensitive learning, and ensemble resampling should be standard pipeline components. Without them, even sophisticated models default to predicting everything as legitimate.

04

Use a Tiered Screening Approach

Use high-recall Logistic Regression as a first-stage screen, followed by a higher-precision model before cases reach human review — capturing the recall strength of the linear model while reducing false positive burden on analysts.

The Recommended Production Architecture

Stage 1 — High-Recall Screen

Logistic Regression flags all suspicious transactions. Catches ~45% of fraud with known false positive cost.

Stage 2 — Precision Filter

Higher-precision model reviews Stage 1 flags and eliminates clear false positives before human review.

Stage 3 — Analyst Queue

Fraud analysts review a manageable, well-prioritized queue. Threshold-tuned to the institution's tolerance.
